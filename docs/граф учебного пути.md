# ТЗ 8. Совместно настраиваемый граф учебного пути

## Продуктовый смысл

Во время реального обучения человек редко движется строго по одному курсу от первой лекции к последней. У него может быть:

- основной курс;
- учебник, который даёт более формальное объяснение;
- отдельное видео для визуальной интуиции;
- статья, закрывающая конкретный пробел;
- notebook для практики;
- собственный проект;
- возврат к prerequisite, который оказался не понят;
- необязательная ветка для более глубокого изучения.

При этом единицей движения должен быть не файл и не урок, а **концепция или способность**.

Например:

```text
Векторы и геометрическая интуиция
→ Линейные преобразования
→ Матрица как представление преобразования
→ Смена базиса
→ Собственные векторы и значения
→ Диагонализация
→ Применение в PCA
```

К каждой концепции могут быть прикреплены разные материалы и действия:

```text
Собственные векторы
├── глава основного курса
├── визуальное видео 3Blue1Brown
├── формальное определение из учебника
├── собственная заметка
├── задача на вычисление
├── объяснение своими словами
└── применение в PCA
```

### Отличие от карты знаний

Эти два графа используют общие концепции, но решают разные задачи.

**Карта знаний** показывает:

- все известные концепции;
- семантические связи;
- текущее состояние понимания;
- пробелы, противоречия и evidence.

**Учебный путь** показывает:

- выбранную цель;
- последовательность движения к ней;
- обязательные и необязательные узлы;
- текущую позицию;
- пройденные и следующие шаги;
- ветки для закрытия пробелов;
- материалы и практику внутри каждого узла;
- причины, по которым маршрут изменился.

Учебный путь — не копия полного knowledge graph. Это выбранная проекция карты знаний под конкретную цель.

### Что означает «совместно настраивать»

Маршрут строится в диалоге пользователя и системы:

1. Пользователь формулирует цель.
2. Система собирает связанные концепции из карты знаний и добавленных материалов.
3. Система предлагает черновой маршрут и объясняет логику порядка.
4. Пользователь подтверждает, переставляет, удаляет или добавляет узлы.
5. Во время обучения система замечает пробелы и предлагает ответвления.
6. Изменение не применяется скрытно: пользователь видит предложение и принимает или отклоняет его.
7. После закрытия пробела временная ветка возвращает пользователя в основной путь.
8. История версий позволяет понять, как и почему маршрут менялся.

На первом этапе «система» может быть детерминированным rule-based planner. LLM-планировщик добавляется позже и использует тот же контракт предложений.

## Готовый промпт для Codex

```markdown
Базовый MVP Personal Learning OS уже работает: есть пространства и цели, материалы, учебные сессии, заметки, концепции, knowledge state, reviews, календарь и экран Today.

Задача: реализовать отдельную продуктовую вертикаль «Учебный путь» — совместно настраиваемый граф движения по концепциям к Learning Goal.

Сначала изучи продуктовые документы и существующие модели LearningGoal, Concept, ConceptRelation, Material, KnowledgeState, CalendarItem. Не создавай вторую независимую модель концепций.

## Основной принцип

Центральные узлы пути — концепции и формируемые способности, а не лекции и файлы.

Материалы, задачи, объяснения и повторения прикрепляются к узлу пути как способы освоить концепцию.

Не смешивай:

- knowledge graph — общая сеть концепций и семантических отношений;
- learning path — выбранный маршрут через часть этой сети под конкретную цель.

## Пользовательский сценарий

1. Пользователь открывает Learning Goal.
2. Нажимает «Построить учебный путь».
3. Система создаёт draft path из существующих концепций и prerequisites.
4. Пользователь видит визуальный граф и список этапов.
5. Пользователь может:
   - добавить существующую концепцию;
   - создать новую концепцию;
   - изменить порядок;
   - соединить узлы;
   - отметить узел обязательным или необязательным;
   - создать альтернативную ветку;
   - прикрепить материал или учебное действие;
   - указать критерий завершения;
   - принять или отклонить предложение системы.
6. После публикации пути текущий узел влияет на Today и Scheduler.
7. При обнаружении prerequisite gap система создаёт предложение временной ветки, но не меняет опубликованный путь без подтверждения.
8. После прохождения ветки маршрут возвращается к исходному узлу.

## Модель данных

### learning_paths

- id UUID PK;
- user_id FK;
- learning_space_id FK;
- learning_goal_id FK;
- title;
- description nullable;
- status: draft | active | paused | completed | archived;
- current_node_id nullable;
- version integer;
- created_at;
- updated_at.

Для одной Learning Goal допускается только один active path. Старые варианты сохраняются как archived или через версии.

### learning_path_nodes

- id UUID PK;
- learning_path_id FK;
- concept_id FK nullable;
- node_type: concept | capability | milestone;
- title;
- description nullable;
- position_x numeric nullable;
- position_y numeric nullable;
- status: planned | available | current | completed | blocked | skipped;
- importance: required | recommended | optional;
- estimated_minutes nullable;
- completion_policy JSONB;
- metadata JSONB default {};
- created_at;
- updated_at.

Для `node_type=concept` поле `concept_id` обязательно. Не дублируй title/description концепции без необходимости: node хранит только контекст конкретного пути.

### learning_path_edges

- id UUID PK;
- learning_path_id FK;
- source_node_id FK;
- target_node_id FK;
- edge_type: sequence | prerequisite | optional_branch | remediation | returns_to;
- condition JSONB nullable;
- label nullable;
- created_at.

Ограничения:

- source != target;
- оба узла принадлежат одному path;
- duplicate edge запрещён;
- sequence/prerequisite graph должен быть acyclic;
- `returns_to` разрешает визуально обозначить возврат из remediation branch, но не должен ломать вычисление доступности.

### learning_path_node_resources

- id UUID PK;
- node_id FK;
- resource_type: material | review_template | practice | explanation | project_task;
- resource_id nullable;
- title;
- is_required boolean;
- order_index integer;
- completion_status: planned | in_progress | completed | skipped;
- metadata JSONB default {};
- created_at;
- updated_at.

### learning_path_suggestions

- id UUID PK;
- learning_path_id FK;
- suggestion_type: add_node | add_branch | reorder | attach_resource | mark_blocked | skip_node;
- payload JSONB;
- rationale TEXT;
- source: rule_engine | knowledge_state | user | llm;
- status: pending | accepted | rejected | expired;
- created_at;
- resolved_at nullable.

### learning_path_versions

- id UUID PK;
- learning_path_id FK;
- version integer;
- snapshot JSONB;
- change_summary;
- change_source: user | accepted_suggestion | system;
- created_at.

Каждое структурное изменение опубликованного active path создаёт новую версию. Draft можно редактировать без версии до первой публикации.

## Completion policy

Узел не считается завершённым только потому, что материал просмотрен.

Поддержи прозрачную policy, например:

```json
{
  "all": [
    {"resource_type": "material", "minimum_completed": 1},
    {"dimension": "explanation", "minimum_score": 60}
  ],
  "any": [
    {"dimension": "application", "minimum_score": 50},
    {"evidence_type": "task_solved", "minimum_count": 1}
  ]
}
```

Для MVP достаточно поддержать документированный поднабор условий. Не создавай универсальный rule language.

## Rule-based path builder v0

Создай planner interface и первую детерминированную реализацию.

Вход:

- Learning Goal;
- выбранные target concepts;
- существующий concept graph;
- prerequisite relations;
- Knowledge State;
- прикреплённые материалы;
- опционально максимальная глубина.

Алгоритм v0:

1. Начать с target concepts цели.
2. Рекурсивно добавить prerequisite concepts до заданной глубины.
3. Не добавлять хорошо освоенный prerequisite как обязательный узел; можно показать его как already known context.
4. Выполнить topological sort обязательной части.
5. Прикрепить известные materials через concept-note/material relations.
6. Для concepts без материалов создать предупреждение, а не выдумывать ресурс.
7. Сформировать draft и rationale для порядка.

Planner должен быть заменяемым. В будущем LLM может предлагать путь, но backend обязан валидировать его теми же domain rules.

## Backend API

Реализуй:

- `POST /api/v1/learning-goals/{goal_id}/path/generate-draft`;
- `POST /api/v1/learning-paths` для пустого ручного path;
- `GET /api/v1/learning-paths/{path_id}`;
- `PATCH /api/v1/learning-paths/{path_id}`;
- `POST /api/v1/learning-paths/{path_id}/publish`;
- CRUD nodes;
- CRUD edges;
- CRUD node resources;
- `POST /learning-paths/{path_id}/nodes/{node_id}/complete`;
- `POST /learning-paths/{path_id}/nodes/{node_id}/skip`;
- `POST /learning-paths/{path_id}/suggestions`;
- `POST /learning-path-suggestions/{id}/accept`;
- `POST /learning-path-suggestions/{id}/reject`;
- `GET /learning-paths/{path_id}/versions`;
- `GET /learning-paths/{path_id}/versions/{version}`;
- `POST /learning-paths/{path_id}/versions/{version}/restore`.

Все структурные изменения выполняй через application use cases. Frontend и AI в будущем не должны напрямую записывать graph tables.

Для конфликтующих параллельных изменений используй optimistic concurrency: клиент передаёт ожидаемую `version`, конфликт возвращает HTTP 409.

## Интеграция с остальной системой

### Today и Scheduler

- current/available required nodes являются главными кандидатами;
- scheduler выбирает действие внутри node resource, а не просто абстрактную concept;
- Today показывает название path node и причину выбора;
- после завершения node current position пересчитывается.

### Knowledge State

- state обновляется из evidence независимо от path;
- path только читает state и использует его в availability/completion;
- завершение path node не должно искусственно выставлять mastery 100.

### Materials

- один material может быть ресурсом нескольких nodes;
- один node может использовать несколько материалов;
- материал не копируется внутрь path.

### Gap detection

Если current node blocked слабым prerequisite:

1. создай suggestion remediation branch;
2. объясни, какой prerequisite мешает;
3. предложи concept/resources/actions;
4. после completion branch верни маршрут через `returns_to`.

## Тесты

### Domain/unit

- запрет cycle;
- topological ordering;
- availability узлов;
- completion policy;
- remediation branch и возврат;
- required/optional semantics;
- optimistic version conflict;
- принятие suggestion атомарно создаёт новую version;
- planner детерминирован на одном входе.

### API

- generate draft;
- ручное редактирование;
- publish;
- complete node;
- accept/reject suggestion;
- restore version;
- isolation по user_id.

### Frontend/E2E

1. открыть goal;
2. сгенерировать draft path;
3. добавить concept node;
4. прикрепить material;
5. опубликовать;
6. увидеть current node на Today;
7. завершить действие;
8. увидеть переход к следующему node;
9. принять remediation suggestion;
10. увидеть новую ветку и запись в version history.

## Критерии приёмки

- путь строится вокруг concepts, а не списка файлов;
- к одному concept node можно прикрепить материалы из разных источников;
- path поддерживает main sequence, optional и remediation branches;
- пользователь может полностью вручную настроить draft;
- система предлагает изменения с rationale, но не применяет их скрытно;
- у active path есть current position и доступные next nodes;
- path интегрирован с Today и Scheduler;
- path version history позволяет посмотреть и восстановить структуру;
- knowledge graph и learning path остаются разными read models;
- все проверки backend/frontend/E2E проходят;
- README и архитектурная документация обновлены.

## Не входит в этот этап

- real-time совместное редактирование несколькими людьми;
- автоматическое применение LLM-плана;
- генерация новых учебных материалов;
- сложная оптимизация пути на ML;
- сравнение эффективности разных пользователей;
- превращение любого course syllabus в идеальный path без подтверждения.

В конце покажи:

1. схему данных;
2. правила availability/completion;
3. пример draft path;
4. пример remediation branch;
5. API contracts;
6. результаты тестов;
7. какие решения оставлены расширяемыми для будущего LLM planner.
```

## Результат этапа

Пользователь получает не просто каталог курсов и не статичную карту знаний, а видимый маршрут через идеи: с текущей позицией, разными источниками, ветками для пробелов, критериями освоения и возможностью совместно перестраивать путь вместе с системой.
